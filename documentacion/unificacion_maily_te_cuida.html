CAMINO DE UNIFICACIÓN

Qué cambiar en Maily te cuida para unificarlo

Lista concreta y accionable de los cambios necesarios para que el backend actual de Maily te cuida (app del paciente) deje de ser un sistema aislado y se integre en la Maily Platform unificada: multi-tenant, identidad global del paciente, especialidades como plugins, autenticación y cobro unificados.

SECCIÓN 1

Objetivo y alcance

El backend actual es de buena calidad (7/10) y está ~80% completo, pero fue construido como una app de paciente aislada (B2C, sin multi-tenant, especialidades fijas, auth con Clerk). Este documento detalla lo que hay que transformar para que sea parte de una sola plataforma donde clínica (Maily Soft) y paciente (Maily te cuida) comparten cimientos.

Recordatorio honesto Unificar es un refactor mayor (semanas, no días). La alternativa —mantener dos backends separados que se comunican por API— es más rápida pero te deja dos sistemas que mantener. Este documento asume que eliges unificar. Si dudas, lee primero la Sección 5 (modelo de negocio) y la 6.

Leyenda de esfuerzo por cambio: BAJO (días) · MEDIO (1-2 semanas) · ALTO (2+ semanas)

CAMBIO 1 · ALTO

C1. Modelo base abstracto + multi-tenant

Es el cimiento de todo. Hoy no hay multi-tenancy: ninguna tabla sabe a qué clínica pertenece. Hay que introducir un modelo base que TODA entidad de negocio herede.

Qué hacer

  1. Crear core/models.py con un modelo abstracto TenantAwareModel.
  2. Migrar cada modelo de negocio para que herede de él (agrega tenant_id, timestamps, soft delete).
  3. Agregar un TenantManager que filtre automáticamente por la clínica del usuario.
  4. Activar Row Level Security en PostgreSQL como segunda barrera.
  5. Middleware que fije el tenant del request a partir del token.
# core/models.py — NUEVO
class TenantAwareModel(models.Model):
    id         = models.UUIDField(primary_key=True, default=uuid4, editable=False)
    tenant_id  = models.UUIDField(db_index=True)        # ← LA clave nueva
    created_at = models.DateTimeField(auto_now_add=True)
    updated_at = models.DateTimeField(auto_now=True)
    deleted_at = models.DateTimeField(null=True, blank=True)

    objects = TenantManager()       # filtra por tenant automático
    all_objects = models.Manager()  # sin filtro (admin)

    class Meta:
        abstract = True
El reto real No es escribir el modelo base (eso es fácil) — es migrar los datos existentes asignándoles un tenant_id y reescribir cada query para respetar el tenant. Por eso es ALTO. Hazlo módulo por módulo, no todo de golpe.
CAMBIO 2 · ALTO

C2. Identidad global del paciente (Master Patient Index)

Hoy PatientProfile es un OneToOne con el usuario y asume una sola relación. En la plataforma, el paciente es global y se conecta a varias clínicas.

Qué hacer

  • Separar la identidad global (PatientAccount, sin tenant) del vínculo por clínica (ClinicPatientLink, con tenant).
  • Mover el perfil de salud "propio del paciente" (alergias, tipo de sangre, crónicos) a la cuenta global.
  • Usar CURP como llave anti-duplicados.
  • Los datos clínicos (vitals, medications, lab_results...) cuelgan del ClinicPatientLink, no del paciente directo.
HOY:   User ──1:1── PatientProfile ──< vitals, medications, ...

OBJETIVO:
  PatientAccount (global, sin tenant)        ← identidad + salud propia + gamificación
        │ se conecta a (1:N)
        ▼
  ClinicPatientLink (con tenant_id)          ← uno por clínica
        │
        ▼
  vitals, medications, lab_results, ...      ← cuelgan del link (tenant-scoped)
Impacto fuerte Esto cambia el "eje" del sistema. Casi todas las apps de datos clínicos (vitals, medications, prescriptions, lab_results, documents) deben repuntar de patient a patient_link. Es de los cambios más invasivos — planéalo con migración de datos y doble lectura.
CAMBIO 3 · MEDIO

C3. Especialidades a sistema de plugins

Hoy nutrición, fisio, etc. son apps Django fijas. En la plataforma deben ser plugins activables por clínica, con su hoja de datos definida por JSON Schema.

Qué hacer

  • Crear un registro de especialidades (Specialty) y un contrato SpecialtyPlugin.
  • Las hojas de cada especialidad se definen por JSON Schema y se guardan en un campo specialty_data (JSONB) del encounter, validado contra el schema.
  • La especialidad se activa por tenant (y es un add-on cobrable).
  • Mantener apps/nutrition como la implementación de referencia del plugin (no tirarla — adaptarla al contrato).
Lo bueno No hay que reescribir la lógica de nutrición; hay que envolverla en el contrato de plugin. El núcleo clínico (encounter común) se comparte, y solo la "hoja" varía por especialidad. Ver el documento de arquitectura, Sección 8.
CAMBIO 4 · MEDIO

C4. Autenticación unificada (Clerk → JWT)

Hoy todo pasa por Clerk (middleware, webhooks, sync de usuarios). La plataforma usa un solo sistema de auth para clínica y paciente.

Qué hacer (2 opciones)

OpciónQué implicaCuándo
A. Migrar a JWT propio (simplejwt)Quitar Clerk, implementar login/registro/refresh propios. Más control, sin dependencia externa, alineado al diseño.Recomendado si Maily Soft también usará JWT propio.
B. Abstraer auth detrás de una interfazDejar Clerk para pacientes pero detrás de una capa auth provider, para poder cambiarlo sin tocar el resto. ~3-4 h de desacople inmediato.Si quieres avanzar ya y decidir después.
Primero, sin importar la opción Corregir YA la validación del JWT de Clerk: hoy no valida iss ni azp (acepta tokens de otras instancias). Es un hallazgo de seguridad ALTO (ver C8).
CAMBIO 5 · MEDIO

C5. Modelo de negocio unificado

Hay un choque: hoy el cobro es B2C mensual (el paciente paga FREE/SILVER/GOLD/PLATINUM); el diseño de la plataforma es B2B anual (la clínica paga + add-ons por especialidad).

Qué hacer: no fusionar, coexistir (cobro DUAL)

FlujoQuién pagaQué
B2B (Maily Soft)La clínicaSuscripción anual + add-ons por especialidad/almacenamiento. Es el ingreso principal.
B2C (Maily te cuida)El paciente (opcional)Premium: IA avanzada, más almacenamiento de estudios, features de bienestar. Ingreso secundario.
  • Renombrar/reubicar el modelo de planes del paciente para que sea claramente "premium del paciente", no el plan de la clínica.
  • El ai_engine que asigna modelo por tier (SILVER→gpt-4o, PLATINUM→claude) se mantiene, pero como beneficio B2C.
  • Unificar la infraestructura de cobro (Stripe) para que ambos flujos usen el mismo motor de suscripciones y webhooks.
Buena noticia Tener dos fuentes de ingreso (clínica + paciente premium) es una ventaja, no un problema. Solo hay que separarlas conceptualmente en el modelo de datos.
CAMBIO 6 · MEDIO

C6. Capa de servicios y selectors

Hoy la lógica vive en views/serializers (consistente, pero fuera del estándar de la plataforma). Hay que migrar gradualmente a la arquitectura por capas de la skill django-clean-architecture.

Qué hacer

  • Por cada app, extraer la lógica de negocio a services.py (escrituras) y selectors.py (lecturas).
  • Las views quedan delgadas: validan entrada, llaman al servicio, devuelven respuesta.
  • Usar como modelo a seguir apps/notifications/service.py y apps/wellness/services.py (ya están bien hechos).
  • Es un cambio gradual: no bloquea, se hace app por app conforme se tocan.
CAMBIO 7 · ALTO

C7. Qué pasa con cada app (la decisión clave)

Varias apps se solapan con lo que hará el backend de la clínica (Maily Soft). Para cada una hay que decidir la fuente de verdad: quién es dueño del dato.

CAMBIO 8 · BAJO · BLOQUEANTE

C8. Correcciones de seguridad (antes de cualquier cosa)

Independientes de la unificación, estos deben corregirse en la primera semana. Son de bajo esfuerzo pero alto impacto:

#ProblemaCorrección
🟠 A1Webhook de MailySoft omite la firma si no hay secreto (usa una variable de entorno inexistente) → se pueden falsificar recetasFallar cerrado: rechazar si no hay secreto; usar la variable ENVIRONMENT correcta; exigir el secreto en producción.
🟠 A2El JWT de Clerk no valida iss ni azp → acepta tokens de otras instanciasValidar issuer y authorized-party; cachear JWKS con TTL.
🟡 M1DSN de Sentry real hardcodeado en base.pyMover a variable de entorno y rotar el DSN.
🟡 M2Django 5.1.4 con CVEs conocidosSubir a la última 5.1.x + pip-audit en CI.
🐞 Bugnotify() no acepta extra_data → TypeError en runtime (notificaciones que nunca se mandan)Corregir la firma + test de regresión.
CAMBIO 9 · BAJO

C9. Calidad y herramientas

  • mypy + django-stubs en CI — cazaría el tipo de bug del extra_data. Type hints en la capa HTTP (hoy casi sin tipar).
  • Modelo base abstracto — ya cubierto en C1; elimina la duplicación de id/created_at/updated_at en cada modelo.
  • Arreglar N+1 — en vitals (una query por tipo de signo vital) y agregar select_related/prefetch_related en listados con FKs.
  • factory_boy + conftest.py — ya está instalado pero no se usa; centralizar fixtures de prueba.
  • pyproject.toml — hoy no existe; agregar config de black, ruff, mypy, pytest en un solo lugar.

SECCIÓN 6

Riesgos

  • Migración de datos: agregar tenant_id e introducir el MPI exige migrar datos existentes con cuidado. Doble lectura y validación cruzada.
  • Alcance: es tentador "arreglar todo de paso". Limítate a las fases; no mezcles refactor de unificación con features nuevas.
  • Romper la app del paciente: hay usuarios en producción (la primera versión). Cada fase debe mantener la app funcionando — feature flags y despliegues graduales.
  • Seguridad olvidada: si C8 no se hace primero, el sistema queda explotable mientras se refactoriza. Es lo primero, sin excepción.
Lo único innegociable Hagas (a) o (b), la Fase 0 (seguridad) se ejecuta YA. El webhook que permite falsificar recetas y la validación del JWT no pueden esperar al refactor.

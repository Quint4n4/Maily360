CONTROL DE ACCESO

Los 3 niveles de roles

🌐 PLATAFORMA
El equipo SaaS (tú). Acceso global, métricas, todos los tenants.
Súper AdminVentas / ÉxitoIngeniería
🏥 TENANT (clínica)
Personal de cada clínica. Aislado por multi-tenant + RLS.
OwnerAdminMédicoEnfermeríaRecepciónFinanzasSolo lectura
🧑 PACIENTE
Identidad GLOBAL (no pertenece a una clínica). Usa Maily te cuida y se conecta a las clínicas del catálogo.
Cuenta global (MPI)

Dos dimensiones independientes: el rol define qué puede hacer (permisos); la especialidad define qué evalúa (su hoja). Una persona puede ser Médico + Nutrición.

El rol nunca cruza niveles: el staff de plataforma no es staff de clínica, y el paciente no está en el árbol de la clínica.

CÓMO SE HABLAN ENTRE SÍ

Comunicación entre módulos — 4 canales

1 · Frontend ↔ Backend
React / Flutter
→ REST →
DRF API

JSON sobre HTTPS con JWT. Un OpenAPI genera los SDK de React y Flutter (tipos siempre sincronizados).

2 · Módulo ↔ Módulo
agenda
→ usa →
pacientes.
services

Se llaman vía la capa de servicios (no tocan las tablas de otro módulo). Cada módulo es dueño de sus datos.

3 · Eventos asíncronos
cita.creada
→
Signals
+
Celery

Una acción emite una señal; otros reaccionan sin bloquear: WhatsApp, bitácora, métricas (Celery + Redis).

4 · Webhooks externos
Stripe / WhatsApp
→ firma →
/webhooks

Sistemas externos notifican al backend (pago, mensaje). Siempre con firma verificada.

Regla de oro: la lógica vive en services/selectors; las vistas son delgadas; los módulos no se acoplan a las tablas de otros.

EL CORAZÓN DEL MODELO

Especialidades como plugins

NÚCLEO COMÚN (85-90%)
auth · tenancy · agenda · pacientes · expediente · recetas · finanzas · facturación · chat · archivos · triage · gamificación
▲ cada especialidad "se enchufa" arriba ▲
🩺 Medicina Generalhoja base (referencia)
🥗 Nutriciónpeso, grasa, plan
🤸 FisioterapiaROM, fuerza, EVA
➕ Ginecología…se agrega sin tocar el núcleo

Cada especialidad define su hoja clínica con un JSON Schema. El backend valida los datos y el frontend genera el formulario solo. Agregar una especialidad nueva = registrar un plugin, sin reprogramar.

Una clínica de 1 especialidad ni ve el concepto de "caso"; una multidisciplinaria activa varias y suma un "Caso" que une las hojas.
AISLAMIENTO + IDENTIDAD

Multi-tenant + identidad global del paciente

👤 PatientAccount — identidad GLOBAL (CURP) · dueño de su cuenta y salud
▼ se conecta a (1 a varias)
🏥 Clínica Agineco · tenant_id
🏥 Clínica Bnutrición · tenant_id
🏥 Clínica Cgeneral · tenant_id

Cada clínica ve SOLO sus propios datos (Row Level Security en PostgreSQL). El paciente ve todo su historial en su app. Para que una clínica vea datos de otra, el paciente da un permiso explícito (SharingGrant).

Patrón estándar de salud (Master Patient Index): identidad compartida, datos clínicos aislados por clínica.

Sin código nuevo. Activar una especialidad es un cambio de datos, no de programa.
Minutos, no semanas. El costo de sumar un cliente es casi cero.
Cobro: plan anual + especialidad como add-on recurrente.
Misma plataforma, misma base de datos. Una fisio, una nutrióloga o una clínica regenerativa: cada quien ve solo sus módulos y su(s) especialidad(es).

EN UNA FRASE

Una plataforma, muchas clínicas, muchas especialidades

Un solo backend multi-tenant sirve a la clínica (Maily Soft), al paciente (Maily te cuida) y al dueño SaaS (Panel). Las especialidades se activan como plugins, los módulos se comunican por servicios y eventos, y vender a un especialista es encender un interruptor — no construir un sistema nuevo.

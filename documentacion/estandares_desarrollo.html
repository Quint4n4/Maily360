EL MODELO DE PROGRAMACIÓN QUE TODOS SEGUIMOS

Estándares de Desarrollo

Las reglas obligatorias para escribir código en Maily Platform con Django REST Framework y Python: arquitectura limpia, tipado fuerte, cero secretos en el código, seguridad y pruebas profesionales. Aplica al equipo interno y a cualquier desarrollador o cliente que extienda la plataforma.

SECCIÓN 1

Filosofía y principios

Escribimos código pensando en quien lo mantendrá dentro de 2 años (que probablemente no seremos nosotros). Estos son los principios no negociables:

🔑

Cero secretos en el código

Ninguna contraseña, llave o token vive en el código ni en git. Todo en variables de entorno o gestor de secretos.

🏷️

Todo tipado

Type hints obligatorios en funciones. mypy en CI. Nada de tipos "any" que se rompen después.

🧱

Capas separadas

Cada cosa en su lugar: modelos, servicios, serializers, vistas. Vistas delgadas, lógica en servicios.

✅

Probado o no existe

Toda lógica de negocio tiene pruebas. Sin tests, el PR no se aprueba.

📏

Formato automático

Black + Ruff formatean e inspeccionan. No discutimos estilo: lo hace la máquina.

📖

Explícito > implícito

Nombres claros, funciones cortas, una responsabilidad cada una. Si necesita comentario para entenderse, se reescribe.

Las 3 reglas que nunca se rompen 1. Jamás un secreto en el código o en git.   2. Jamás SQL armado a mano con datos del usuario (siempre el ORM).   3. Jamás lógica de negocio dentro de una vista o un serializer.
SECCIÓN 2

Arquitectura de capas

Seguimos una arquitectura de capas inspirada en el Django Styleguide de HackSoft (el estándar de oro en la comunidad). Cada capa tiene un único trabajo y solo puede hablar con la capa de abajo:

  PETICIÓN HTTP
       │
       ▼
  ┌─────────────────────────────────────────────────────────────┐
  │ URLs        →  ruteo (qué endpoint llama a qué vista)         │
  ├─────────────────────────────────────────────────────────────┤
  │ VIEWS       →  DELGADAS. Reciben request, llaman al servicio  │
  │ (DRF)          o selector, devuelven respuesta. SIN lógica.   │
  ├─────────────────────────────────────────────────────────────┤
  │ SERIALIZERS →  validan ENTRADA y dan forma a la SALIDA.       │
  │                NO contienen lógica de negocio.                │
  ├─────────────────────────────────────────────────────────────┤
  │ SERVICES    →  ESCRIBEN. Toda la lógica de negocio (casos de  │
  │                uso): crear cita, registrar consulta, cobrar.  │
  │ SELECTORS   →  LEEN. Consultas complejas (queries) a la BD.   │
  ├─────────────────────────────────────────────────────────────┤
  │ MODELS      →  datos + validación propia (clean). Managers    │
  │                para queries reutilizables. SIN casos de uso.  │
  └─────────────────────────────────────────────────────────────┘
       │
       ▼
  PostgreSQL
CapaSÍ haceNO hace
View / ViewSetParsear request, llamar a 1 servicio/selector, devolver respuesta y código HTTPNo consultas a BD, no reglas de negocio, no cálculos
SerializerValidar formato de entrada, dar forma a la salida (JSON)No guardar en BD, no lógica de negocio, no efectos secundarios
ServiceCasos de uso que modifican datos: crear, actualizar, procesosNo tocar el request HTTP, no serializar
SelectorConsultas de lectura complejas, con filtros y permisosNo modificar datos
ModelDefinir datos, validación propia (clean()), propiedades simplesNo casos de uso completos, no llamadas a servicios externos

Ejemplo: crear una cita (el flujo correcto)

# views.py — DELGADA
class AppointmentCreateApi(APIView):
    class InputSerializer(serializers.Serializer):
        patient_id = serializers.UUIDField()
        starts_at  = serializers.DateTimeField()

    def post(self, request) -> Response:
        s = self.InputSerializer(data=request.data)
        s.is_valid(raise_exception=True)
        appointment = appointment_create(            # ← delega al servicio
            tenant=request.tenant, user=request.user, **s.validated_data
        )
        return Response(AppointmentSerializer(appointment).data, status=201)

# services.py — la lógica vive aquí
def appointment_create(*, tenant: Clinic, user: User,
                       patient_id: UUID, starts_at: datetime) -> Appointment:
    patient = patient_get(patient_id=patient_id)        # selector
    if _slot_taken(tenant, starts_at):
        raise ValidationError("Ese horario ya está ocupado")
    appointment = Appointment.objects.create(
        tenant=tenant, patient=patient, starts_at=starts_at, created_by=user,
    )
    appointment_created.send(sender=Appointment, appointment=appointment)  # señal
    return appointment

Settings separados por ambiente, apps por dominio de negocio (no por tabla), cada app autocontenida y "enchufable".

│   │   ├── dev.py          ← desarrollo

Regla de apps Cada app = un dominio (agenda, pacientes, finanzas), no una tabla. Una app debe poder explicarse en una frase ("maneja todo lo de la agenda"). Si no, está mal dividida.
SECCIÓN 4

Convenciones de código

TemaRegla
FormateoBlack (formato automático) + Ruff (linter rápido). Corren en pre-commit y CI. No se discute estilo.
ImportsOrdenados por Ruff/isort: stdlib → terceros → locales. Sin imports sin usar.
NombresVariables y funciones en snake_case; clases en PascalCase; constantes en MAYÚSCULAS. En inglés, descriptivos.
ServiciosFunciones nombradas como acción: appointment_create, patient_update. Argumentos por nombre (keyword-only con *).
FuncionesCortas y con una responsabilidad. Si pasa de ~40 líneas o tiene varios niveles de anidación, se parte.
DocstringsEn servicios y funciones públicas no triviales: qué hace, qué recibe, qué devuelve, qué excepciones lanza.
ComentariosExplican el porqué, no el qué. Si el código necesita un comentario para entender qué hace, se reescribe.
ErroresExcepciones específicas (ValidationError, PermissionDenied), nunca except: a secas.
SECCIÓN 5

Tipado obligatorio

Esto resuelve tu preocupación de "tipado feo que se rompe después". Reglas:

  • Type hints obligatorios en TODA firma de función (argumentos y retorno). Los locales se infieren, no se anotan salvo que sea ambiguo.
  • django-stubs + mypy con plugin de Django, corriendo en CI. Un PR con errores de tipo no se mergea.
  • Activamos los checks valiosos sin volver loco al equipo: disallow_untyped_defs, disallow_any_generics, warn_return_any, warn_unused_ignores.
  • Prohibido Any "por flojera". Si de verdad se necesita, se justifica en el PR.
# ❌ MAL — sin tipos, devuelve cualquier cosa
def get_patient(id):
    return Patient.objects.get(pk=id)

# ✅ BIEN — tipado explícito y claro
def patient_get(*, patient_id: UUID) -> Patient:
    return Patient.objects.get(id=patient_id)

# pyproject.toml — config de mypy
[tool.mypy]
plugins = ["mypy_django_plugin.main"]
disallow_untyped_defs = true
disallow_any_generics = true
warn_return_any = true
warn_unused_ignores = true
SECCIÓN 6

Reglas de Django REST Framework

TemaRegla
Serializers de entrada y salida separadosUn InputSerializer para validar lo que llega y otro OutputSerializer para dar forma a lo que sale. No reusar el mismo para todo.
Serializers delgadosSolo validan y dan forma. Nada de create()/update() con lógica de negocio — eso va a servicios.
Campos mínimosIncluir solo los campos necesarios en la salida (payloads ligeros, mejor para móvil).
AutenticaciónJWT con djangorestframework-simplejwt. Stateless, escala horizontal.
PermisosClases de permiso por endpoint. Multi-tenant + rol. Devolver 404 (no 403) cuando el recurso existe pero no es del usuario (no revelar su existencia).
Consultas eficientesselect_related / prefetch_related siempre que haya relaciones. Cero queries N+1.
PaginaciónObligatoria en todo listado. Nunca devolver "todo".
ThrottlingLímite de peticiones por usuario/IP para proteger la API.
VersionadoRutas con versión: /api/v1/.... Romper compatibilidad = nueva versión.
Documentacióndrf-spectacular genera el OpenAPI automático (del que salen los SDK de React y Flutter).
SECCIÓN 7

Secretos y configuración

Esta sección resuelve directamente tu requisito de "nada hardcodeado". Es la regla #1.

# ❌ PROHIBIDO — el error del sistema legacy
SECRET_KEY = "m@1LYmx2o2oo6o1"
DATABASES = {"default": {"PASSWORD": "clave123"}}

# ✅ CORRECTO — leído del entorno, nunca en git
import environ
env = environ.Env()
SECRET_KEY = env("DJANGO_SECRET_KEY")          # 50+ caracteres aleatorios
DATABASES = {"default": env.db("DATABASE_URL")}
DEBUG = env.bool("DJANGO_DEBUG", default=False)
ReglaDetalle
Nada de secretos en código ni en gitNi en repos privados. Si un secreto se filtra, se rota de inmediato.
Variables de entornoCon django-environ. En local: archivo .env (ignorado por git). Se versiona solo .env.example con nombres, sin valores.
Gestor de secretos en producciónAWS Secrets Manager (o Vault). Rotación automática, auditoría y acceso controlado.
SECRET_KEYMínimo 50 caracteres aleatorios, generado con get_random_secret_key().
git-secrets / pre-commitHook que bloquea el commit si detecta algo que parezca una credencial.

Seguridad (checklist OWASP)

Basado en el OWASP Django Security Cheat Sheet. Es obligatorio antes de cualquier despliegue a producción:

ControlRegla
DEBUG = False en producciónNo negociable. En debug se filtran trazas y datos sensibles.
ALLOWED_HOSTS explícitoSolo los dominios reales, para evitar ataques de Host header.
HTTPS + HSTSSECURE_HSTS_SECONDS > 0, cookies Secure + HttpOnly + SameSite.
SQL: siempre el ORMNunca concatenar datos del usuario en SQL. (Lo que rompió al legacy.)
Contraseñas hasheadasEl hasher de Django (Argon2/PBKDF2). Jamás cifrado reversible.
Multi-tenant + RLSCada query filtra por tenant; doble barrera con Row Level Security en Postgres.
Bitácora de auditoríaRegistro inmutable de accesos y cambios a expedientes (NOM-024).
Permisos por defecto: denegarDEFAULT_PERMISSION_CLASSES = [IsAuthenticated]. Lo abierto se abre a propósito.
Dependencias auditadaspip-audit / Dependabot en CI para detectar librerías con vulnerabilidades.
Validación de entradaToda entrada se valida en el serializer antes de tocar la lógica.
SECCIÓN 9

Testing profesional

"Probado o no existe." Toda lógica de negocio (servicios y selectors) tiene pruebas. Stack: pytest + pytest-django + factory_boy.

Qué se pruebaCómo
Servicios (prioridad #1)Cada caso de uso: camino feliz + casos de error + reglas de negocio.
SelectorsQue filtren bien por tenant y permisos (que NO se filtren datos de otra clínica).
APIsCódigos HTTP correctos, validación de entrada, permisos (401/403/404).
ModelosValidaciones propias (clean()) y métodos.
  • Cobertura mínima: 80% en código de negocio (servicios/selectors), medida en CI. El PR que la baje no se mergea.
  • Patrón AAA (Arrange-Act-Assert): preparar datos, ejecutar, verificar.
  • factory_boy para crear datos de prueba (no fixtures gigantes a mano).
  • Las pruebas son rápidas y aisladas (no dependen del orden ni de internet).
# test_services.py
def test_appointment_create_rejects_double_booking(db):
    # Arrange
    clinic = ClinicFactory()
    AppointmentFactory(tenant=clinic, starts_at="2026-06-01T10:00Z")
    # Act + Assert
    with pytest.raises(ValidationError):
        appointment_create(tenant=clinic, user=UserFactory(),
                           patient_id=PatientFactory().id,
                           starts_at="2026-06-01T10:00Z")

Git y CI/CD

TemaRegla
Ramasmain siempre desplegable. Trabajo en ramas feature/..., fix/.... Nada se sube directo a main.
CommitsConventional Commits: feat:, fix:, refactor:, test:, docs:.
Pull RequestsPequeños y enfocados. Descripción de qué y por qué. Mínimo 1 revisión aprobada.
pre-commit hooksAntes de cada commit corren: Black, Ruff, mypy, detección de secretos. Si fallan, no hay commit.
Pipeline CI (GitHub Actions)En cada PR: lint → mypy → tests → cobertura → pip-audit. Todo verde o no se mergea.
MigracionesRevisadas en el PR. Reversibles. Nunca editar una migración ya aplicada en producción.
SECCIÓN 11

Definition of Done (checklist de PR)

Un cambio está "terminado" solo si cumple TODO esto. Es la lista que revisan los agentes antes de aprobar:

  ☐ La lógica de negocio está en services/selectors, no en views ni serializers
  ☐ Todas las funciones tienen type hints; mypy pasa sin errores
  ☐ Black + Ruff pasan; sin imports ni código muerto
  ☐ Cero secretos en el código (lo verifica el hook)
  ☐ Serializers de entrada/salida separados y delgados
  ☐ Queries optimizadas (select_related/prefetch_related); sin N+1
  ☐ Permisos y filtrado por tenant verificados (no se fugan datos)
  ☐ Pruebas para los casos de uso nuevos; cobertura ≥ 80% en negocio
  ☐ Migraciones reversibles y revisadas
  ☐ Pasa el checklist OWASP relevante
  ☐ PR pequeño, con descripción de qué y por qué
  ☐ CI 100% verde

Skill: django-clean-architecture

La "fuente de verdad" de estos estándares en formato que Claude usa al programar. Se activa sola al tocar código Django.

4 agentes especializados

Cada uno experto en su rol, aplicando este documento.

AgenteQué haceCuándo se usa
django-engineerEscribe código Django/DRF siguiendo esta arquitectura: tipado, capas, sin secretos.Al implementar una feature o módulo nuevo.
django-reviewerRevisa que el código cumpla TODO este documento (la Definition of Done).Antes de aprobar un PR.
django-testerEscribe pruebas profesionales (pytest + factory_boy), busca cobertura y casos borde.Después de implementar, o cuando falta cobertura.

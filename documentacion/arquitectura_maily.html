SECCIÓN 1

Resumen ejecutivo

Estamos transformando un sistema legacy de gestión de clínicas (escrito en PHP) en una plataforma SaaS moderna y multi-tenant. La plataforma da servicio a tres tipos de usuarios desde un solo backend:

  • El doctor y su personal → trabajan en Maily Soft (la app web de la clínica).
  • El paciente → usa Maily te cuida (app móvil): ve su expediente, chatea con su doctor, recibe recordatorios, guarda estudios, gana recompensas y puede preguntar a una IA a qué especialista acudir según sus síntomas.
  • Tú, el dueño del SaaS → tienes un Panel de Plataforma con métricas de todo: ingresos, suscripciones, clínicas activas, adopción de funciones, etc.
La idea central No son tres sistemas separados, son tres caras de la misma plataforma. Comparten la misma base de datos y el mismo "cerebro", por eso el expediente que escribe el doctor es exactamente el que ve el paciente, sin desincronización.

SECCIÓN 5

Multi-tenant: la columna vertebral

Multi-tenant significa que muchas clínicas distintas (los "tenants") viven en el mismo sistema, pero cada una está totalmente aislada de las demás. Es como un edificio de departamentos: misma estructura, pero nadie entra al departamento de otro.

Hay dos niveles claramente separados:

🌐 GLOBAL (nivel plataforma, SIN dueño-clínica) — solo 3 cosas:
   • PatientAccount   ← la persona (su cuenta de Maily te cuida)
   • Doctor           ← identidad profesional (puede trabajar en varias clínicas)
   • Specialty        ← catálogo de especialidades

🏥 TENANT (por clínica, con aislamiento garantizado por la base de datos):
   • Clinic, suscripción, billing
   • Owner / Admin / Médico / Enfermería / Recepción ... (el árbol de roles)
   • ClinicPatientLink  ← el vínculo paciente ↔ esta clínica
   • ClinicalEncounter  ← las consultas (cada clínica ve SOLO las suyas)
   • Recetas, agenda, finanzas, inventario, archivos...
Garantía técnica: Row Level Security (RLS) PostgreSQL aplica un candado automático: aunque un programador se equivoque, la base de datos solo devuelve las filas de la clínica del usuario que pregunta. Es imposible que la Clínica A vea datos de la Clínica B.

Importante: "rol" y "especialidad" son dos cosas distintas

Una persona del personal tiene dos atributos independientes que no se deben confundir:

  • Rol = qué puede hacer en el sistema (sus permisos). Ej: Médico, Enfermería, Recepción.
  • Especialidad = qué evalúa y qué hoja clínica llena. Ej: Nutrición, Fisioterapia, Medicina Regenerativa.

Por eso un mismo usuario puede ser, a la vez, Rol: Médico + Especialidad: Nutrición. En una clínica con varias especialidades, cada especialista entra con su propio rol y su propia especialidad:

Ana López     →  Rol: Médico         +  Especialidad: Nutrición
Beto Ruiz     →  Rol: Médico         +  Especialidad: Fisioterapia
Dra. Pérez    →  Rol: Médico/Owner   +  Especialidad: Medicina Regenerativa
Recep. Sara   →  Rol: Recepción      +  (sin especialidad)
Por qué se separan Los permisos (rol) cambian poco; las especialidades varían mucho y se cobran como add-on. Mantenerlas separadas permite que una clínica agregue una especialidad nueva sin rehacer todo el esquema de permisos.

Las 3 reglas que esto impone

  1. El paciente ve TODO su historial (de todas las clínicas) en Maily te cuida, porque es dueño de su cuenta.
  2. Cada clínica ve SOLO sus propios encounters por defecto (aislamiento multi-tenant).
  3. El paciente puede COMPARTIR su historial de una clínica con otra mediante un consentimiento explícito (SharingGrant).
CURP = llave anti-duplicados Si una clínica da de alta manualmente a "Juan Pérez" y Juan ya tiene cuenta en Maily te cuida, el sistema los vincula por CURP en vez de crear dos pacientes distintos.

El beneficio comercial Cada especialidad es un add-on cobrable. Una clínica solo carga, ve y paga las especialidades que usa. El sistema no se hace pesado para el cliente, aunque por dentro sea una sola base de código.

El "contrato" de especialidad (ejemplo en código)

Cada especialidad implementa un mismo contrato. Agregar una nueva especialidad no requiere tocar el núcleo.

# Medicina General — la del REPOSITORIO, primera (referencia)
class GeneralMedicinePlugin(SpecialtyPlugin):
    code = 'general_medicine'
    name = 'Medicina General'
    def encounter_schema(self):
        return {}   # solo usa el núcleo tipado, sin campos extra

# Nutrición — la SEGUNDA, de prueba
class NutritionPlugin(SpecialtyPlugin):
    code = 'nutrition'
    name = 'Nutrición'
    def encounter_schema(self):
        return {  # JSON Schema → valida datos y genera el formulario solo
            "type": "object",
            "properties": {
                "peso_kg":           {"type": "number"},
                "porcentaje_grasa":  {"type": "number"},
                "masa_muscular_kg":  {"type": "number"},
                "kcal_objetivo":     {"type": "integer"},
                "plan_alimenticio":  {"type": "string"},
            },
            "required": ["peso_kg"]
        }

Caso real: varios especialistas evaluando al MISMO paciente en la MISMA clínica

Esto es opcional y aditivo — el plan de una sola especialidad sigue igual La mayoría de las clínicas contratan una sola especialidad (ej. un consultorio de nutrición o de ginecología) y operan con la hoja simple de esa especialidad: nunca ven el concepto de "Caso". La colaboración multidisciplinaria que se describe abajo solo se activa cuando una clínica tiene 2 o más especialidades (cada una contratada como add-on). Es la misma plataforma; solo se "encienden" más especialidades.

Hay clínicas (por ejemplo, de medicina regenerativa) donde un mismo paciente es evaluado por nutrición, fisioterapia y el médico tratante, y todas esas evaluaciones se juntan para decidir el protocolo de tratamiento. Esto pasa dentro de una sola clínica (un solo tenant), con varios especialistas colaborando.

Tu intuición es correcta: cada especialista llena su propia "hoja de especialidad" — que no es más que el formulario que se genera solo desde el JSON Schema de su especialidad. Encima de eso, agregamos una pieza que lo amarra todo: el "Caso" (o Episodio), que agrupa las hojas de todos los especialistas y produce la decisión consensuada.

🏥 CLÍNICA REGENERATIVA  (1 tenant · varias especialidades activadas)
│
└── 🧑 Paciente Juan  (ClinicPatientLink)
       │
       └── 📋 CASO: "Lesión de rodilla — terapia regenerativa"
              coordinador: Dra. Pérez (Medicina Regenerativa)
              estado: EN EVALUACIÓN → PROTOCOLO DEFINIDO → EN TRATAMIENTO → ALTA
              │
              ├── 🥗 Hoja de NUTRICIÓN        → Lic. Ana
              │      peso, % grasa, marcadores de inflamación,
              │      plan alimenticio antiinflamatorio...
              │
              ├── 🤸 Hoja de FISIOTERAPIA     → Lic. Beto
              │      rango de movilidad, fuerza, escala de dolor,
              │      plan de ejercicios...
              │
              ├── 🩺 Hoja MÉDICA REGENERATIVA → Dra. Pérez
              │      diagnóstico, estudios de imagen,
              │      indicación de terapia celular / PRP...

                     "Plan integral: dieta antiinflamatoria + fisio 3x/sem

Cómo se resuelve, en simple Cada especialista solo llena su hoja (la de su especialidad, ni más ni menos). El sistema las junta automáticamente bajo un mismo Caso ligado al paciente. El médico coordinador ve todas las hojas en un solo lugar y captura el protocolo. Así la decisión multidisciplinaria queda registrada y trazable: se sabe quién evaluó qué y por qué se eligió ese tratamiento.
PiezaQué esQuién la llena
Caso / EpisodioEl "expediente del tratamiento": agrupa todas las evaluaciones de un mismo objetivo clínico.Lo abre el médico coordinador (o recepción).
Hoja de especialidadLa evaluación de cada especialista (formulario propio de su especialidad). Técnicamente es un encounter ligado al caso.Cada especialista la suya.
ProtocoloLa decisión final consensuada a partir de todas las hojas.El médico coordinador.
Lo mejor: no requiere nada "especial" de programación Esto reusa exactamente lo que ya teníamos (especialidades como plugins + hojas por JSON Schema). Solo agregamos la entidad Caso que las agrupa. Sirve igual para una clínica regenerativa, un equipo de rehabilitación, o un comité médico (tumor board en oncología). Es un patrón estándar: atención multidisciplinaria.
SECCIÓN 9

Modelo de negocio (cómo se cobra)

Modelo de negocio: suscripción ANUAL No hay mensualidad. La clínica contrata por un año completo (un solo cobro), precedido de un periodo de prueba gratis para que conozca el sistema antes de pagar. Esto da ingresos predecibles, reduce la cancelación mes a mes y mejora el flujo de caja.

El cobro funciona en 3 capas que conviven: un plan base anual, add-ons opcionales, y un control manual tuyo para casos especiales.

# ══════ NIVEL PLATAFORMA — GLOBAL (sin tenant_id) ══════

class PatientAccount:        # el ser humano + login de Maily te cuida
    curp, first_name, last_name, birth_date, email, phone
    blood_type, allergies, chronic_conditions   # perfil de salud propio
    points, level                               # gamificación

    user, cedula, full_name, specialties[], is_in_catalog

class Specialty:             # catálogo: general_medicine, nutrition...
    code, name, plugin_class, is_billable

# ══════ NIVEL TENANT — POR CLÍNICA (con tenant_id + RLS) ══════

class Clinic:                # = Tenant
    name, slug, specialties[]

class ClinicPatientLink:     # vincula paciente global ↔ clínica
    patient, numero_expediente, fecha_alta, origen, estado

class StaffSpecialty:         # qué especialidad(es) ejerce un miembro
    membership, specialty       # Ana = Médico + Nutrición

class TreatmentCase:          # "Caso/Episodio" multidisciplinario
    patient_link, titulo        # "Lesión de rodilla — regenerativa"
    coordinator                 # médico que lidera el caso
    status                      # evaluacion/protocolo/tratamiento/alta
    protocolo                   # decisión consensuada (texto/JSON)

class ClinicalEncounter:     # la consulta = "hoja de especialidad"
    patient_link, doctor, specialty, fecha
    case                        # ← a qué Caso pertenece (puede ser null)
    motivo_consulta, signos_vitales, diagnostico_cie10, plan  # núcleo
    specialty_data  # JSONB validado por especialidad

# ══════ CONTROLADO POR EL PACIENTE ══════

class SharingGrant:          # el paciente comparte historial entre clínicas
    patient, source_clinic, grant_to_clinic, expires_at, revoked_at

   Django + drf-spectacular
            │
            ├─► genera  openapi.yaml  (el "contrato" de la API)
            │
   ┌────────┼──────────────────────┐

Cambias un campo en el backend → se regeneran los SDKs → las 3 apps se enteran del cambio y dejan de compilar si algo no cuadra. Imposible que la app del paciente se desincronice del expediente del doctor.

SECCIÓN 12

Flujos de ejemplo (cómo viajan los datos)

Flujo A — El paciente usa el triage con IA y se conecta a una clínica

1. Paciente (Flutter): "me duele el bajo vientre, tengo retraso"
        │ POST /api/v1/triage/analyze
        ▼
2. apps/triage  →  Claude clasifica → especialidad = "ginecología"
        ▼
3. apps/directory  →  busca (global) doctores con specialty=gineco,
                      en el catálogo, con cupo en agenda
        ▼
4. Respuesta: "Sugerimos Ginecología. Dra. X (Clínica A) tiene
              cita mañana 10am"   [orientación, NO diagnóstico ⚠]
        │ paciente acepta
        ▼
5. POST /api/v1/me/connect-clinic { clinic_id, doctor_id }
        ▼
6. apps/patients  →  crea ClinicPatientLink (vincula por CURP)
        ▼
7. apps/agenda  →  crea cita  →  señal "cita.creada"
        ├─► Celery: WhatsApp de confirmación al paciente
        └─► WebSocket: notifica a la clínica en Maily Soft

Flujo B — El doctor de nutrición registra una consulta

1. Doctor (React): abre expediente, pestaña "Nutrición"
        │ GET /api/v1/specialties/nutrition/schema
        ▼
2. Backend devuelve el JSON Schema → React PINTA EL FORMULARIO solo
   (peso, % grasa, pliegues, kcal objetivo...)
        │ doctor llena y guarda
        ▼
3. POST /api/v1/encounters { specialty: nutrition, signos_vitales,
                             specialty_data: {peso_kg, grasa, ...} }
        ▼
4. apps/clinical  →  valida specialty_data CONTRA el schema de nutrición
                     →  guarda en clinical_encounters (tenant = Clínica B)
        ├─► señal "encounter.creado" → bitácora (NOM-024)
        └─► el paciente lo ve al instante en Maily te cuida

Flujo C — El paciente comparte su historial entre clínicas

La nutrióloga (Clínica B) quiere ver qué recetó la ginecóloga (Clínica A)
        ▼
1. Paciente (Flutter): "Compartir mi historial de Clínica A con Clínica B"
        │ POST /api/v1/me/sharing-grants { source: A, grant_to: B }
        ▼
2. apps/sharing  →  crea SharingGrant
        ▼
3. Ahora la nutrióloga ve los encounters de Clínica A SOLO porque
   existe el permiso. Sin permiso → no los ve.

Glosario (para entender los términos)

TérminoQué significa en simple
SaaS"Software como Servicio". Vendes el uso del software por suscripción mensual, no una licencia.
Multi-tenantMuchas clínicas usan el mismo sistema pero cada una está aislada, como departamentos en un edificio.
TenantCada clínica cliente. Su "departamento" privado dentro del sistema.
RLS (Row Level Security)Candado de la base de datos que garantiza que una clínica solo vea sus propias filas.
Master Patient Index (MPI)El paciente es una sola identidad global aunque visite varias clínicas.
BackendEl "cerebro" del sistema: lógica y base de datos. No se ve, pero hace todo.
FrontendLas pantallas que el usuario ve y toca (web o app).
APIEl "menú" de funciones que el backend ofrece a las apps para pedir o guardar datos.
Bounded contextUn área de negocio bien delimitada (ej. "finanzas", "agenda") con su propia lógica.
JSONBUna columna flexible de la base de datos que guarda datos con estructura variable (ideal para especialidades).
Plugin / verticalUn módulo "enchufable" que agrega una especialidad sin tocar el núcleo.
Add-onFunción extra que se compra encima del plan base.
Feature flagUn "switch" para encender o apagar una función a clientes específicos.
MRR / ARRIngreso recurrente mensual / anual. La métrica más importante de un SaaS.
ChurnPorcentaje de clientes que cancelan. Mientras más bajo, mejor.
TriageClasificar al paciente según sus síntomas para mandarlo al especialista correcto.
NOM-024Norma mexicana que regula el expediente clínico electrónico.
CURPClave única de población en México. Sirve para no duplicar pacientes.
